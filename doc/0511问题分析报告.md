# P0：必须优先修复的问题

## 1. Pro 权益可被客户端伪造

位置：

- `deploy_package/publish.php:210-229`
- `ios/CloudService.swift:125-126`

当前逻辑：

```php
$client_is_pro = isset($_POST['is_pro']) && $_POST['is_pro'] === '1';

if (!$is_pro && $client_is_pro) {
    $is_pro = true;
    $db->execute("... ON DUPLICATE KEY UPDATE is_pro = 1 ...");
}
```

iOS 客户端直接上传：

```swift
let isProUser = SubscriptionManager.shared.isPro
body.append(formField(name: "is_pro", value: isProUser ? "1" : "0", boundary: boundary))
```

问题：

- 服务端“声明不信任客户端”，但实际完全信任了客户端传来的 `is_pro=1`
- 任何人只要知道 API Key，就可以伪造 Pro
- 免费用户限制、自定义短链、永久链接等都可绕过

影响：

- **商业化模型失效**
- **付费权益可被绕过**
- **App Store 购买状态与服务端权益不一致**

建议：

- 服务端必须接入 Apple App Store Server API / Receipt Verification
- `users.is_pro` 只能由服务端验证交易后更新
- [publish.php](cci:7://file:///Volumes/ssd/aicode_new0421/ioscode/zaixianhtml/backend/publish.php:0:0-0:0) 不应根据客户端 `is_pro` 自动升 Pro
- iOS 只传 `transaction_id/original_transaction_id` 或服务端 token，不直接传权益结果

优先级：**P0**

`CE_HMAC_SECRET_KEY` 未出现在 [Info.plist](cci:7://file:///Volumes/ssd/aicode_new0421/ioscode/zaixianhtml/ios/Info.plist:0:0-0:0)，因此客户端会 fallback 到 API Key 作为 HMAC secret：

```swift
let secret = hmacSecret.isEmpty ? apiKey : hmacSecret
```

问题：

- API Key 硬编码在 App 包中
- HMAC secret 如果 fallback 到 API Key，安全性更弱
- 一旦 key 泄露，所有 API 可被调用

建议：

- 短期：至少增加 `CE_HMAC_SECRET_KEY`，并与 API Key 分离
- 中期：使用服务端换取短期 token
- 长期：用户身份系统 + 服务端签发 JWT / session token

优先级：**P1**

# 五、安全与合规专项评估

## 当前安全优点

- HMAC + timestamp 防重放
- 文件路径有 sanitize / safe join
- 上传扩展名白名单
- 文件大小限制
- 数据库使用 PDO prepared statements
- 密码使用 bcrypt
- 禁止目录浏览
- 禁止访问 [.env](cci:7://file:///Volumes/ssd/aicode_new0421/ioscode/zaixianhtml/deploy_package/.env:0:0-0:0)

## 主要安全缺陷

- **API Key 在客户端，可被逆向**
- **Pro 权益可伪造**
- **项目管理 API 缺少所有权校验**
- **CORS 部分接口 `*`**
- **IP 原文存储**
- **管理后台安全性需要进一步审计**
- **发布内容是用户 HTML/JS，天然可执行任意脚本**

关于用户 HTML：

发布的 HTML 由用户提供，访问者打开后会执行其中 JS。由于域名统一为 `html.niceapp.eu.cc`，如果以后加入用户登录 cookie 或后台 cookie，需要注意跨项目脚本隔离。

建议：

- 发布内容使用独立无 Cookie 子域，例如：
  - `usercontent.html.niceapp.eu.cc`
- 管理/API 使用：
  - `api.html.niceapp.eu.cc`
- 设置严格 Cookie domain，不让用户 HTML 读取管理 cookie

- 功能覆盖全面

  - 本地项目删除后云端是否还存在？

# 十一、最终结论

这个项目的基础很好，尤其是 iOS 端功能密度很高，已经不只是简单 HTML 编辑器，而是“HTML 编辑 + 发布托管 + 分享管理”的完整产品雏形。

但当前最大的风险不是功能数量，而是 **生产一致性和安全闭环**：

- [/deploy_package](cci:9://file:///Volumes/ssd/aicode_new0421/ioscode/zaixianhtml/deploy_package:0:0-0:0) 生产代码与部分修复代码不同步
- DB schema 与代码依赖不一致
- Pro 权益可伪造
- 云端管理 API 越权
- 短链访问资源路径可能存在体验问题

如果先修 P0/P1，项目可以很快进入稳定 TestFlight / 早期上线状态。  
如果不修，正式上线后最可能出现的问题是：

- 用户改过期时间无效
- 访问记录页面空/不完整
- 免费用户绕过 Pro
- 他人项目被越权操作
- 新服务器部署后 SQL 字段报错
- 短链打开 HTML 但 CSS/JS 丢失
